# ADR-0004: `defineResponsePlugin` 响应后处理策略

## 背景

`@hile/http` 的 `defineController` 原本在控制器执行后，直接将返回值写入 `ctx.body`。随着 HTTP 输出场景变复杂（统一响应包裹、按请求类型转换输出、特定路径定制化等），需要一个统一且可组合的“响应后处理层”。

如果这些逻辑分散在各个控制器中，会导致：

- 响应格式策略不一致
- 控制器职责膨胀，业务代码与协议层耦合
- 跨路由策略（如统一 envelope）难以复用

## 决策

在 `@hile/http` 中引入并采用 `defineResponsePlugin` 作为控制器结果的统一后处理机制：

1. 控制器返回值先进入响应插件链，再决定最终 `ctx.body`。
2. 插件函数签名为：
   - `(ctx, result, next) => Promise<any>`
3. 插件按注册顺序串行执行，形态为 onion-like 的 `next(transformedResult)` 传递。
4. 只有插件链最终传递结果 `!== undefined` 时，才写入 `ctx.body`。
5. 插件能力定位在“结果变换层”，不替代 Koa 全局中间件职责。

对应代码行为要点：

- `defineController` 在控制器函数执行后调用 `composeResponsePlugin(...)`
- 最终以插件链输出结果 `r` 作为 `ctx.body`（而非原始 `result`）

## 影响

正向：

- 统一响应策略可集中管理、可复用
- 控制器保持“返回业务结果”的单一职责
- 便于扩展内容协商/包装/过滤等能力

代价：

- 插件链引入额外执行开销（通常可接受）
- 插件顺序会影响最终结果，需要治理与约定
- 若插件误传 `undefined`，可能导致无响应体，需要测试覆盖

## 约束与实践规则

1. 插件应保持纯结果变换语义，避免承载业务流程。
2. 插件必须显式调用 `next(...)` 传递结果。
3. 需要“吞掉响应体”时，必须有明确注释与测试。
4. 对公共响应格式插件，建议在框架初始化阶段集中注册。

## 备选方案

1. **在每个控制器里手动包装返回值**
   - 缺点：重复代码多，难统一演进。
2. **仅使用 Koa 全局中间件处理 body**
   - 缺点：难稳定拿到“控制器原始返回值语义”，并且职责边界不清。
3. **在 `defineController` 内置单一固定包装规则**
   - 缺点：扩展性差，无法满足多策略并存。
